# 审计手册

按类别告诉你该看什么。每个子智能体（或直接审计环节）会收到对应章节加上文末的**Finding format**。根据仓库规模调整深度，一个 2K 行的 CLI 得到的扫描强度，自然低于一个 50 万行的 monorepo。

一条发现要成立，必须有证据。"某处可能存在 N+1 查询"不叫发现；`orders/api.ts:142 在循环中为每个订单项发起一次查询` 才是。

---

## 1. 正确性 / Bug

可信度最高的类别，真正的 Bug 要靠读代码发现，而非靠猜测。

- 错误处理：被吞掉的异常、空 catch 块、关键路径上的 `catch (e) { console.log(e) }`、UI 代码中缺失的错误状态。
- 异步隐患：未 await 的 Promise、共享状态上的竞态、缺失的取消/清理（React effect 中的过期闭包、永远没被移除的监听器）。
- 空值流：可能为 null 的值上的非空断言（`!`）、为"必须存在"的值使用可选链、未做检查的数组下标。
- 边界条件：差一错误、空集合处理、时区/区域假设、计数器/ID 中的整数溢出。
- 状态机：类型上能表达的不可能状态组合、存在未处理分支的状态枚举（注意那些"静默 no-op"的 `default:`）。
- 并发：对共享资源的"先检查再操作"、多次写入操作缺少事务、重试操作的幂等性（webhook、队列）。
- 类型逃逸出口：成簇出现的 `any` / `as` 断言 / `@ts-ignore`，每一处都是编译器被驳回的地方。
- 资源泄漏：未关闭的句柄、连接、订阅；缺少 `finally`。

## 2. 安全

只审核那些能由代码证据直接支撑的内容。把发现框定为"防御性维护"：指出代码模式，解释对生产的影响，并描述修复方案。计划停留在代码改动、配置改动和测试层面；不要包含可运行的演示字符串或一步步的误用说明。

**处理规则：** 绝不要把密钥值复制到发现或计划中，那些文件是要提交的。只引用 `file:line` 和凭据类型（例如 "Stripe live key at `config.ts:12`"），修复草图始终要包含轮换，而不仅仅是删除（已提交的密钥即便删除也已经暴露了）。

**"设计如此"不是发现：** 标准平台约定属于有意为之，尊重 `https_proxy`/`NO_PROXY`、读取 `~/.netrc`、某个明确的本地开发工具去调用已配置的包管理器。在 ADR 或决策文档中明文记录的权衡同样属"已定下"，不是发现。只有当**实现本身**在约定或已记录决策之外引入了额外风险时，才予以标记，同时注意**过时的 ADR 本身也是一条发现**：如果代码已经偏离决策文档所说的内容，请报告"决策漂移"（文档或代码有错；无论如何团队都该知道），不要用文档去压制它。

- 凭据卫生：硬编码的 key/token/password、提交进仓库的 `.env` 文件中的凭据、被记录或持久化到事件/历史存储中的凭据。发现中只应说明凭据类型和位置，然后建议删除、轮换并改用更安全的配置方式。
- 进入解释器或特权 API 的数据：用请求数据拼装的 SQL/Shell 操作（SQL/命令注入）、接收用户可控内容的 HTML 接收点（XSS）、接受运行时输入的动态执行 API，或由请求数据派生的文件系统路径（路径穿越）。描述更安全的 API 或校验边界；不要给出可运行的示例。
- 访问控制：缺乏服务端身份校验的端点/server action、仅在客户端执行的授权、通过 ID 访问对象而没有归属或租户校验（IDOR）、或对状态变更路由缺少请求真实性校验（CSRF）。
- 输入契约：API 边界在没有 schema 校验的情况下信任请求体、上传文件时缺少明确的类型/大小/存储限制、或把请求数据广泛赋值到持久化模型（批量赋值）。
- 依赖姿态：以只读模式运行生态的审计命令（`npm audit`、`pip-audit`、`cargo audit`）。只报告影响可达运行时代码或构建/分发路径的 critical/high 公告；避免低信噪比的审计噪声。
- 生产配置：允许凭据时过于宽松的 CORS、在涉及敏感浏览器面时缺失响应加固头部（例如 CSP）、cookie 缺少适当的 `HttpOnly`/`Secure`/`SameSite` 属性、生产配置中开启 debug/verbose 行为。
- 数据最小化：日志中的 PII 或敏感运营数据、返回给客户端的堆栈跟踪、通过 API 响应暴露的内部错误细节。

## 3. 性能

寻找算法和架构层面的胜利，而非微优化。

- N+1 模式：循环内或每行渲染中的逐项查询/获取；缺少批处理或 dataloader。
- 错误的复杂度：在同一集合上嵌套扫描、在热循环中重复 `find`/`filter`，而本该用 Map 做键查找。
- 缓存缺口：相同且昂贵的计算或获取在每次请求/渲染中重复；缺少在清晰函数边界的记忆化；稳定数据上无 HTTP/数据层缓存。
- 有效载荷大小：过量获取（select *、在只需 ID 时取完整对象）、在无界列表上缺少分页、把大块 JSON 推给客户端。
- 前端（如适用）：包体构成（为微小用途引入重型依赖）、对低频路由缺少代码分割、未优化的图片/字体、应可在渲染时获得的数据却走客户端获取、渲染瀑布。对 React/Next.js，遵循仓库的框架约定和任何已安装的"最佳实践"指南。
- 后端：本应在队列中完成的同步工作、由查询模式暗示但缺失的索引（标记为待验证，没有 schema 证据不要断言）、在已有连接池的情况下每请求一个连接的模式。
- 构建/CI：因缺少缓存而慢的 CI、冗余的流水线步骤、可并行化但没有并行的测试套件。

## 4. 测试覆盖

目标不是百分比，而是**哪些未测试的代码是危险的**。

- 绘制关键路径（钱、鉴权、数据变更、仓库存在的那个特性），看哪些是零覆盖或仅有名义覆盖。
- 高 churn（git log）+ 无测试 = 顶级重构风险；标记为"先做刻画测试"候选。
- 已有测试的质量：断言不出任何有意义内容的测试、为测 mock 而 mock 的重 mock 模式、没人读的快照测试、易抖动模式（真实计时器、真实网络、顺序依赖）。
- 缺失的测试层：只有单元测试、API 边界却没有任何集成覆盖，或反之（用慢的 E2E 测本可以由单元测试搞定的东西）。
- 验证基础设施：有没有一条命令就能知道代码库是好的？如果没有，那就是第 1 条发现，也是任何有风险改动的前置计划。

## 5. 技术债与架构

- 重复：同一逻辑在 3+ 处被重新实现（搜索近乎相同的函数/组件）；已经发生偏离的多个副本。
- 分层违规：UI 直接 import 数据层内部、循环依赖、"utils"模块变成高扇入的杂物抽屉。
- 死代码：未导出且未使用的模块、特性标志已全量开启但仍在分支的代码、没有任何解释的注释掉代码块、清单中列出但已不再被 import 的依赖。
- 上帝对象/模块：文件大小是仓库中位数的数量级、且人人都会触及；参数两位数或条件嵌套很深的函数。
- 不一致的写法：同一个仓库里有三种数据获取/错误处理/样式的方式，选出"赢家"（团队最近收敛到的那种），并规划整合。
- 抽象不匹配：只有一种实现的过早抽象，或缺少抽象导致同一个改动总是要同时改 N 个文件。

## 6. 依赖与迁移

- 核心框架/运行时的主要版本滞后（不是每个次要 bump，那些有真实代价的：EOL、安全修复截止、生态不兼容）。
- 仍在使用、但已公布移除时间线的已弃用 API。
- 关键路径上的已废弃依赖（多年无发布、仓库已存档）。
- 解决同一问题的重复依赖（两个日期库、两个 HTTP 客户端）。
- 锁文件/清单漂移、monorepo 中版本钉的不一致。
- 对每个候选迁移，估算爆炸半径（涉及的文件数），即这决定成本与是否值得推荐。

## 7. DX 与工具链

- 缺失或坏掉的：typecheck 脚本、lint 配置、格式化器、pre-commit 钩子、editorconfig。
- 慢反馈环：以分钟计的 dev-server 或测试启动时间、没有 watch 模式、CI 没有缓存。
- 上手摩擦：README 中的安装步骤是错的或不完整、未文档化的必需环境变量、缺少 `.env.example`。
- 缺少 `CLAUDE.md`/`AGENTS.md`，对于要让智能体执行计划的仓库，这是高杠杆的：建议补一份，并把大纲作为一项计划。
- 错误信息/日志：服务侧的非结构化日志、缺少 request ID/关联 ID、调试时需要改代码。

## 8. 文档

默认优先级最低，只在缺失会带来具体代价时标记：

- 已发布包的公共 API 表面缺少参考文档。
- 没人能复原的架构决策（为什么是 X 而不是 Y），即出现在活跃争议区域。
- 那些"明显错误"的过期文档（比没有更糟），即安装说明、不再能编译的 API 示例。

## 9. 方向，下一步做什么功能

前瞻性：不是哪里坏了，而是这个代码库想变成什么。**锚定规则：** 每条建议都必须引用仓库自身的证据，那种"放到任何此类项目都成立"的建议（"加个深色模式"、"加个 AI"）是噪声，不是发现。锚定方向信号的来源：

- **未完成意图**：围绕同一主题的 TODO/FIXME 簇、从未真正发布的特性开关、半成品或被注释掉的功能代码、git 历史中可见的中途放弃的工作。
- **已声明但未交付**：README/文档/路线图中的承诺没有对应代码、no-op 的 CLI flag 或配置选项、为不存在特性留的 issue 模板。PRD 或 `PRODUCT.md` 写明了用户、场景或方向而代码还没跟上，这是最强的锚定信号；优先于推断出的意图，绝不要提出决策文档已经拒绝的方案（改为记录这个矛盾）。
- **面的不对称**：单向配对（只有 export 没有 import、只有 create 没有 bulk-create、只有出方向没有入方向的 webhooks）、实体有 CRUD 但缺一条、公开 API 显然是内部代码所需要并被手卷绕过的。
- **可邻近性**：现有架构使之变得相对廉价的能力，离一套插件系统只差一个接口的现状、离一个公开 API 只差一个路由文件的服务层、早已被数据模型支持的一种集成。
- **值得产品化的摩擦**：项目的用户显然在仓库之外手工做着的事情（可见于文档、示例、issues），项目本可以吸收掉。

方向发现使用标准格式，但有两点调整：**影响**指产品/用户价值（谁想要这个、为什么是现在），**置信度**反映证据的"锚定程度"，而非"这是正确决定"的确定性。战略属于维护者；顾问的工作是给出有依据的选项和诚实的权衡。此处的成本估算更粗略；要明说。被选中的方向发现对应的计划通常是 *设计/探索性计划*（调研、原型、定义 API、列出开放问题），而不是"一锅端"的构建计划，按这个口径来范围。

## 发现格式

每条发现，来自任何类别、任何子智能体，都按下面的形态返回：

```markdown
### [CATEGORY-NN] 简短的祈使句标题

- **证据**：`path/file.ts:123`，一句话描述该位置的情况。（如有多处则重复列；2–5 处最强位置；广泛存在时注明"另有 ~N 处类似"。）
- **影响**：因为这个会出现什么问题 / 在付出什么代价。要具体，例如"每次订单列表渲染发起 1+N 次查询"，而不是"存在次优"。
- **成本**：S（数小时）/ M（一天左右）/ L（多日），指**修复**本身，含测试。
- **风险**：修复可能破坏什么；LOW/MED/HIGH，外加一行原因。
- **置信度**：HIGH（读代码已确定）/ MED（强信号，待验证）/ LOW（仅是气味，需调查）。LOW 置信度发现可以报告，但对应的是"调查"计划，而非"修复"计划。
- **修复草图**：1–3 句。不是完整计划，够用来诚实判断成本即可。
```

## 排序准则

按**杠杆值 = 影响 ÷ 成本，按置信度与修复风险折扣**排序。决胜规则：

1. 能解锁其他发现的事项（验证基线、刻画测试）上浮。
2. HIGH 置信度的安全发现上浮到同等杠杆值的非安全发现之上。
3. 偏向"修复有清晰验证路径"的发现，执行者模型擅长这些。
4. "不值得做"是合法结论；用一行理由记录下来，让用户知道它被考虑过。
