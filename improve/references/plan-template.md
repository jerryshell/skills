# 交接计划模板

每份计划都写给一个**零上下文**的执行者模型：它没有见过顾问会话、没见过审计、没见过其他计划、也没见过任何先前对话。它可能是一个更小、更便宜的模型。假设它"听话照做"是合格的，但补缺、容忍歧义、知道何时停下都是它的弱项。

三个特性让计划能被弱模型执行：

1. **自包含的上下文**，文件内包含执行所需的全部内容：路径、代码摘录、约定、命令。
2. **验证关卡**，每一步都以一条命令和预期结果结尾。执行者永远不需要*判断*自己是否成功。
3. **硬边界与应急出口**，明确的范围外清单，以及"STOP 并报告"条件，而不是放任模型在现实与计划不符时即兴发挥。

文件命名：`plans/NNN-short-slug.md`，按推荐执行顺序编号。

---

## 模板

```markdown
# Plan NNN: <祈使句标题，计划完成后什么会变成真>

> **执行者说明**：按本计划一步步执行。运行每一条验证命令，在确认符合预期结果后再进入下一步。如果"STOP 条件"一节中的任何情况发生，请停下并报告，不要即兴发挥。完成后，更新 `plans/README.md` 中本计划对应的状态行，除非调你的人告诉你他会维护索引。
>
> **漂移检查（先跑）**：`git diff --stat <planned-at SHA>..HEAD -- <in-scope paths>`
> 如果范围内文件自本计划撰写以来发生变化，请将"当前状态"摘录与最新代码对比再继续；若不匹配，视为 STOP 条件。

## Status

- **Priority**: P1 | P2 | P3
- **Effort**: S | M | L
- **Risk**: LOW | MED | HIGH
- **Depends on**: plans/NNN-*.md（或 "none"）
- **Category**: bug | security | perf | tests | tech-debt | migration | dx | docs | direction
- **Planned at**: commit `<short SHA>`, <YYYY-MM-DD>

## Why this matters

2–5 句。问题是什么、它带来的具体代价，以及落地后会得到什么改善。让执行者（和人类审阅者）理解意图，当细节跑偏时，意图才是正确判断的依据。

## Current state

执行者需要的事实，内联，永远不要写"如上所述"或"见审计"：

- 相关文件，每个附一行说明其角色：
  - `src/orders/api.ts`，订单列表端点；包含 N+1（第 130–160 行）
- 现有代码的摘录（要短，附 `file:line` 标记），足以让执行者确认自己看对了位置。
- 本仓库适用的约定，附一个示例文件：
  "错误处理遵循 Result 模式，见 `src/lib/result.ts` 及其在 `src/users/api.ts:40-60` 中的用法。照着做。"
- 计划必须遵循的任何已文档化词汇或设计约束，内联自侦察中找到的意图/设计文档：执行者应在命名与注释中使用的 `CONTEXT.md` 词汇、应复用的 `DESIGN.md` token/组件，或本工作必须保持一致的 ADR。引用具体行，执行者没读过这些文档。

## Suggested executor toolkit

（可选，只有在执行者环境中"很可能存在"相关技能/工具时填写。否则跳过本节。）

- 执行者应该按需调用的技能，以及用途：
  "在第 3 步写记忆化时，使用 `vercel-react-best-practices`"。
- 动手前值得一读的参考文档，按路径或 URL 给出。

## Scope

**In scope**（你应修改的唯一文件）：
- `src/orders/api.ts`
- `src/orders/api.test.ts`（新建）

**Out of scope**（不要碰，即便看起来相关）：
- `src/orders/legacy-api.ts`，已废弃路径，计划删除；改它只会浪费精力并影响仍钉在 v1 上的客户端。
- 对外响应形态的任何变更，客户端依赖于此。

## Git workflow

（按侦察结果填写，匹配仓库中观察到的约定。）

- 分支：`advisor/NNN-<slug>`（或仓库中明显的命名约定）
- 提交粒度：每步一次或每个逻辑单元一次；消息风格：<匹配仓库，例如 conventional commits，引用 `git log` 中的一个示例>
- 除非运营者明确指示，不要 push 或开 PR。

## Steps

### Step 1: <祈使句标题>

要做什么，写得精确。引用确切的文件/符号。当"目标代码形态"是承重的，附上（要产出的模式，不一定逐行）。

**Verify**: `<command>` → <预期输出>

### Step 2: ...

（每一步要小到能独立验证。尽量让步骤之间的顺序保证代码库始终不破，例如：先加新路径、再切调用方、最后删旧路径。）

## Test plan

- 要写哪些新测试、在哪个文件、覆盖哪些情形（列出：正常路径、本计划要修的特定 Bug/回归、命名的边界情形）。
- 拿哪份现有测试做结构模板：
  "以 `src/users/api.test.ts` 为模板"。
- 验证：`<test command>` → 全部通过，含 N 个新测试。

## Done criteria

机器可校验。**所有**项必须满足：

- [ ] `pnpm typecheck` 退出码为 0
- [ ] `pnpm test` 退出码为 0；针对 <X> 的新测试存在并通过
- [ ] `grep -rn "<old pattern>" src/` 无任何匹配
- [ ] 范围外列表之外没有文件被修改（`git status`）
- [ ] `plans/README.md` 状态行已更新

## STOP conditions

停下来并报告（不要即兴发挥）当：

- "当前状态"中位置处的代码与摘录不匹配（自本计划撰写以来代码已发生漂移）。
- 某步的验证在合理修复尝试后仍连续失败两次。
- 修复看起来需要触碰范围外的文件。
- 你发现假设"<关键假设>"是假的。

## Maintenance notes

给改动落地后负责这段代码的人/智能体：

- 未来什么变会与本改动交互（例如"如果给本端点加上分页，第 2 步的批处理必须重审"）。
- 复审 PR 时应当重点看什么。
- 任何被本计划明确推迟的后续事项（及原因）。
```

## 索引文件：`plans/README.md`

由顾问在所有计划写完后写一次，由执行者更新：

```markdown
# Implementation Plans

由 improve 技能于 <date> 生成。除非依赖关系另有说明，否则按下表顺序执行。每位执行者：开始前完整阅读计划，遵守 STOP 条件，完成后更新自己的行。

## Execution order & status

| Plan | Title | Priority | Effort | Depends on | Status |
|------|-------|----------|--------|------------|--------|
| 001  | ...   | P1       | S      | 无         | TODO   |
| 002  | ...   | P1       | M      | 001        | TODO   |

Status 取值：TODO | IN PROGRESS | DONE | BLOCKED（附一行原因） | REJECTED（附一行理由，发现已被独立修复或方案被放弃）

## Dependency notes

- 002 依赖 001，因为 <reason>。

## Findings considered and rejected

- <finding>：不值得做，因为 <一行>。（这样下次就没人重新审计它了。）
```

## 质量门槛，每写完一份计划都过一遍

- 一个从未见过本仓库的模型，仅凭这份计划文件和仓库本身，能否执行？如果有一步依赖了顾问会话里的知识，把那个知识内联进来。
- 每一条验证都是带预期结果的命令，而不是判断（"确保它能工作"）？
- 每一步都点明了确切的文件和符号，而不是"相关模块"？
- STOP 条件是针对本计划实际风险的，不是套话？
- 一个审阅者只读 "Why this matters" + "Done criteria"，能理解自己批准的是什么吗？
- 文件中没有任何密钥，只有位置和凭据类型。
- "Planned at" SHA 已填，漂移检查中的 in-scope 路径与 Scope 一节相匹配。
